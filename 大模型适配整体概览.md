## 执行摘要

1. **"大模型适配"不是单一动作，而是一个六层技术栈**：硬件层 → 框架层 → 模型结构层 → 训练/微调层 → 服务/部署层 → 应用/场景层。任何一次"把大模型用起来"的工作，本质上都是在这六层中的某几层做"对齐"。明确自己在哪一层、与哪一层接壤，是一切适配工作的起点。

2. **适配的本质是"四个对齐"**：与**硬件**对齐（算子/精度/显存）、与**任务**对齐（领域知识/指令遵循）、与**人类偏好**对齐（有用/无害/诚实）、与**业务系统**对齐（接口/延迟/成本/合规）。技术栈再复杂，最终都服务于这四个对齐。

3. **国产化适配已从"能跑"进入"全链路"阶段**：2025 年初 DeepSeek 爆发后，昇腾、沐曦、天数、摩尔线程、海光等十余家国产芯片厂商在一周内完成训练到推理的全链条适配；2026 年 DeepSeek-V4 技术报告首次正式写入昇腾 NPU，标志硬件适配从"迁移可用"走向"原生协同设计"。([来源](https://www.doit.com.cn/p/528158.html), [来源](https://www.nbd.com.cn/articles/2026-04-24/4358607.html))

4. **"训练侧适配"的主线是 PEFT + 对齐**：全参微调让位于参数高效微调（LoRA/QLoRA/Prefix/Prompt Tuning），对齐则由 RLHF（奖励模型 + PPO）走向更轻量稳定的 DPO 等直接偏好优化路线。([来源](https://www.cnblogs.com/KubeExplorer/p/18674820))

5. **"推理侧适配"的主线是量化 + 显存/KV 管理 + 分布式服务化**：2026 年 W4A8KV4（4-bit 权重、8-bit 激活、4-bit KV Cache）成为许多团队的默认配置，FP8 由 H200/B200/MI350 原生支持；PagedAttention 把显存利用率推到 96%+；Dynamo、llm-d、AIBrix、KServe 等把推理引擎搬上 Kubernetes 做规模化服务。([来源](https://zhuanlan.zhihu.com/p/2017695401436783377), [来源](https://www.cnblogs.com/zackstang/p/19036108))

6. **"应用侧适配"的主线是 RAG → Agentic RAG → Agent**：从提示工程到检索增强，再到具备规划、工具调用与自我反思能力的智能体；幻觉治理需要 RAG + CoT + RLHF 多技术协同（斯坦福研究显示组合方案可使幻觉率较基座降低约 96%）。([来源](https://zhuanlan.zhihu.com/p/1981389972620735200), [来源](https://blog.csdn.net/2401_84204413/article/details/153867077))

**核心结论**：成功的大模型适配 = **"先定位分层，再选型路线，最后用数据与评测闭环驱动迭代"**。不要在错误的层解决问题（例如用提示工程硬解本该靠微调解决的领域能力缺失，或用微调硬解本该靠 RAG 解决的知识时效问题）。**选对层，比在某一层用力更重要。**

# 第 0 章　导论：什么是"大模型适配"

## 0.1 概念边界

"大模型适配"（LLM Adaptation）在中文语境中是一个**广义的伞形术语**，至少包含四类语义，容易混用：

| 语义 | 通俗说法 | 关键问题 | 本书所在层 |
|---|---|---|---|
| **硬件适配** | "让模型在某芯片上跑起来" | 算子、精度、显存、互联 | 第 3 章 |
| **框架/引擎适配** | "让模型在某推理/训练框架里跑得快" | 算子映射、调度、并行 | 第 4、5 章 |
| **任务/领域适配** | "让模型懂我的行业、听我的指令" | 知识、风格、指令遵循 | 第 6 章 |
| **应用/系统适配** | "让模型嵌入我的业务流程" | 接口、知识库、工具、延迟 | 第 7、8 章 |

本书的立场：**这四类不是互相替代，而是一条自底向上的栈。** 一个真实项目往往要同时触碰其中多层。把它们放在同一张地图上看，是理解"适配"的关键。

## 0.2 为什么"适配"成为独立学科

- **模型与硬件解耦失败**：大模型对显存、带宽、互联、精度的要求远超传统模型，"训练能用的卡"和"推理划算的卡"经常不同，催生了专门的硬件适配工程。
- **通用 ≠ 可用**：基座模型（Foundation Model）是"通才"，但企业要的是"在我的数据、我的合规、我的延迟预算下可用的专才"，这中间的全部落差，就是"适配"要填的。
- **成本是第一性约束**：适配的每一个选择（量化位宽、是否微调、是否上 Agent）最终都折算为**单位 token 成本 × 延迟 × 质量**的三角权衡。

## 0.3 阅读方式

- 想建立全局观 → 读第一篇（第 1–2 章）。
- 负责某一层 → 直接读第二篇对应章，再回看第 12–13 章看它在流程中的位置。
- 做技术选型/方案 → 重点读第四篇（第 12–14 章）。

# 第一篇　全景：适配的分层模型

# 第 1 章　六层适配栈总览

大模型适配可以抽象为自底向上的**六层栈**。每一层都有自己的"输入—适配动作—输出"，下层为上层提供能力，上层向下层提出约束。

```
┌─────────────────────────────────────────────────────────────┐
│  L6  应用 / 场景适配层   Prompt · RAG · Agent · 工作流编排      │  ← 面向业务价值
├─────────────────────────────────────────────────────────────┤
│  L5  服务 / 部署适配层   量化 · KV管理 · 批处理 · 服务化 · 路由   │  ← 面向延迟/吞吐/成本
├─────────────────────────────────────────────────────────────┤
│  L4  训练 / 微调 / 对齐层  CPT · SFT · PEFT · RLHF/DPO          │  ← 面向能力/偏好
├─────────────────────────────────────────────────────────────┤
│  L3  模型结构适配层      架构改造 · 算子重写 · 并行切分           │  ← 面向可运行/可扩展
├─────────────────────────────────────────────────────────────┤
│  L2  框架 / 引擎适配层    训练框架 · 推理引擎 · 编译器           │  ← 面向"在哪跑"
├─────────────────────────────────────────────────────────────┤
│  L1  硬件 / 芯片适配层    GPU/NPU · CANN/CUDA · 显存 · 互联      │  ← 面向算力底座
└─────────────────────────────────────────────────────────────┘
            横向贯穿：数据工程 · 评测/可观测/LLMOps · 安全合规成本
```

| 层 | 核心问题 | 典型产物 | 典型角色 |
|---|---|---|---|
| **L1 硬件** | 这张卡能不能、值不值得跑这个模型？ | 算子库、驱动、精度支持 | 芯片/系统工程师 |
| **L2 框架** | 用什么引擎把模型跑起来？ | 训练框架、推理引擎配置 | 平台/推理工程师 |
| **L3 模型结构** | 模型本身要不要改造才能高效运行？ | 改写后的模型定义、并行策略 | 模型/Kernel 工程师 |
| **L4 训练微调** | 怎么让模型获得新能力/新偏好？ | 微调权重、对齐模型 | 算法工程师 |
| **L5 服务部署** | 怎么又快又省地对外提供服务？ | 量化模型、推理服务、API | MLOps/推理工程师 |
| **L6 应用场景** | 怎么把模型变成业务价值？ | RAG/Agent 应用 | 应用/产品工程师 |

> **关键认知**：大多数"适配失败"，不是某层做得不够好，而是**在错误的层投入**。第 13 章的决策树专门解决"该在哪层动手"。

# 第 2 章　适配的四个"对齐"维度

六层栈是"结构视角"，"四个对齐"是"目标视角"。所有适配动作最终都在做下面四件事之一：

1. **硬件对齐（Hardware Alignment）**——让模型的数据类型、算子、显存占用、通信模式与底层算力匹配。对应 L1–L3、L5 的量化部分。核心张力：*精度 vs. 速度/显存*。

2. **任务对齐（Task Alignment）**——让模型掌握领域知识、遵循特定指令、输出特定格式。对应 L4 的 CPT/SFT 与 L6 的 RAG/提示。核心张力：*注入新知识 vs. 灾难性遗忘*。

3. **偏好对齐（Preference / Value Alignment）**——让模型有用（Helpful）、无害（Harmless）、诚实（Honest），符合人类与监管期望。对应 L4 的 RLHF/DPO 与 L6 的护栏。核心张力：*能力 vs. 安全（对齐税 Alignment Tax）*。

4. **系统对齐（System Alignment）**——让模型符合业务系统对接口、延迟 SLA、并发、可用性、成本、审计的要求。对应 L5–L6 与横向的 LLMOps。核心张力：*质量 vs. 成本/延迟*。

> **用法**：拿到任何一个适配需求，先问"这主要是四个对齐里的哪一个？"——答案直接决定你该进哪一层。

# 第二篇　纵向各层概览

# 第 3 章　硬件与芯片适配层（L1）

## 3.1 这一层解决什么

把"模型计算图"落到"具体算力硬件"上，并回答两个问题：**能不能跑（功能正确）** 与 **划不划算（性能/成本）**。

## 3.2 全景：算力底座的两条生态

| 生态 | 代表 | 软件栈 | 适配特点 |
|---|---|---|---|
| **英伟达主导生态** | A100/H100/H200/B200 | CUDA + cuDNN + NCCL + Triton | 生态最成熟，是事实标准，适配成本最低 |
| **国产/异构生态** | 昇腾 910B/910C、沐曦、天数智芯、摩尔线程、海光、寒武纪 | CANN（昇腾）、各家自有 SDK | 自主可控，但需逐模型/逐算子适配 |

华为昇腾构建了"**芯片 + MindSpore 框架 + Atlas 服务器**"全栈生态，以 CANN（Compute Architecture for Neural Networks）提供从算子开发到模型部署的全栈工具；CloudMatrix 384 超节点支持 384 张昇腾芯片高速互联，集群算力达 ~300 PFLOPS。([来源](https://zhuanlan.zhihu.com/p/637918406))

## 3.3 国产化适配现状（2025–2026）

- **2025 年初**：DeepSeek 爆发，**7 天内 16 家国产芯片厂商**（昇腾、沐曦、天数智芯、摩尔线程、海光等）完成对 DeepSeek 的适配或上架，覆盖训练到推理全链条。([来源](https://www.doit.com.cn/p/528158.html))
- **2026 年 4 月**：DeepSeek-V4 预览版技术报告**首次正式写入昇腾 NPU 与英伟达 GPU**，标志国产硬件从"被动迁移"迈向"原生协同设计"。([来源](https://www.nbd.com.cn/articles/2026-04-24/4358607.html))

## 3.4 硬件适配的关键工作项（不展开实现）

- **算子覆盖**：模型用到的算子是否在目标芯片算子库中有高性能实现；缺失的需自定义算子（custom op）。
- **精度支持**：是否原生支持 FP16/BF16/FP8/INT8；FP8 由 NVIDIA H200/B200、AMD MI350 原生支持并使吞吐翻倍。([来源](https://zhuanlan.zhihu.com/p/2017695401436783377))
- **显存与带宽**：单卡显存能否容纳权重 + KV Cache；HBM 带宽往往是推理瓶颈。
- **互联与拓扑**：多卡/多机的 NVLink/私有高速互联，决定张量/流水并行的效率。
- **驱动与工具链成熟度**：编译器、profiler、debugger 是否齐全，直接影响适配工时。

> **决策要点**：硬件适配是"是否值得"的经济学问题。评估维度 = 算子覆盖率 × 实测性能 × 生态成熟度 × 供应/合规约束。

# 第 4 章　框架与推理引擎适配层（L2）

## 4.1 这一层解决什么

在硬件之上选择"运行时"：**训练用什么框架，推理用什么引擎**。框架把模型定义、并行策略、内核调度、内存管理封装起来。

## 4.2 训练侧框架

| 框架 | 定位 |
|---|---|
| PyTorch（+ FSDP/DeepSpeed/Megatron-LM） | 事实标准，分布式训练主力 |
| MindSpore | 昇腾深度优化，国产全栈 |
| JAX/PaddlePaddle 等 | 特定生态 |

## 4.3 推理侧引擎（本书重点接壤面）

| 引擎 | 核心特点 |
|---|---|
| **vLLM** | PagedAttention（借鉴操作系统虚拟内存分页），KV Cache 分页管理，显存利用率 96%+，高吞吐服务化标杆 ([来源](https://www.cnblogs.com/zackstang/p/19036108)) |
| **TensorRT-LLM** | NVIDIA 官方，FP8/INT8 KV Cache，极致单机性能 |
| **SGLang** | RadixAttention，结构化生成/复杂调度 |
| **LMDeploy** | INT8/INT4 KV Cache，国产部署常用 |
| **HF Transformers / TGI** | 通用、易用，支持到 INT2 KV Cache 量化 ([来源](https://zhuanlan.zhihu.com/p/2017695401436783377)) |

## 4.4 引擎适配的关键工作项

- **模型加载与权重映射**：新架构模型能否被引擎识别、权重 layout 是否兼容。
- **量化格式对接**：引擎支持的量化方案（GPTQ/AWQ/FP8/W4A8 等）。
- **并行策略**：张量并行（TP）、流水并行（PP）、专家并行（EP，MoE 必备）。
- **调度与批处理**：连续批处理（continuous batching）、前缀缓存、投机解码支持度。
- **API/协议**：是否提供 OpenAI 兼容接口，便于上层无缝替换。

# 第 5 章　模型结构适配层（L3）

## 5.1 这一层解决什么

当"现成框架 + 现成硬件"不能直接高效跑某个模型时，需要在**模型结构层面**做适配：要么改模型以贴合硬件，要么写内核以支撑模型。

## 5.2 典型适配动作（概览）

- **算子重写/融合**：把多个小算子融合（如 FlashAttention 系列），减少访存。
- **架构改造适配**：MoE（专家并行/路由）、长上下文（位置编码外推、注意力变体）、多模态（视觉/音频编码器对接）。
- **并行切分**：把模型沿张量/流水/专家/序列维度切分到多卡多机。
- **图编译/优化**：torch.compile、编译器把动态图转为高效静态执行。
- **数值稳定性**：低精度下的溢出/下溢处理、归一化位置调整。

> **与本书读者的接壤**：在 vLLM 这类引擎中"适配一个新模型"，本质就是 L2+L3 的工作——实现该模型的前向定义、注册量化与并行支持、对接 KV/采样接口。本书不展开具体某模型的实现，但定位清晰：**它属于"模型结构 × 引擎"的交叉层。**

# 第 6 章　训练 / 微调 / 对齐适配层（L4）

这是"任务对齐"与"偏好对齐"的主战场，也是"算法工程师"语境下最常说的"大模型适配/二次开发"。

## 6.1 适配强度阶梯（从轻到重）

```
提示工程  <  RAG  <  PEFT 微调  <  全参微调  <  持续预训练  <  从头预训练
  低成本 ─────────────────────────────────────────────→ 高成本
  改"怎么问" → 改"给什么知识" → 改"模型权重一部分" → 改"全部权重" → 重塑底层知识
```

> **方法论核心**：能用低强度解决就不用高强度。先提示→再 RAG→再微调→最后才考虑持续预训练。

## 6.2 持续预训练 / 领域自适应预训练（CPT / DAP）

- **目的**：在通用基座上，用**领域无标注语料（金融/法律/医疗/代码，10B–100B tokens 级别）**继续预训练，注入领域知识与术语范式。([来源](https://developer.aliyun.com/article/1684045))
- **核心风险**：**灾难性遗忘（Catastrophic Forgetting）**——学新忘旧。需通过混合通用语料、回放（replay）、学习率控制缓解。([来源](https://blog.csdn.net/sinat_37574187/article/details/144110788))
- **何时用**：领域与通用语料差异巨大、且有大规模领域语料时。否则优先 SFT/RAG。

## 6.3 监督微调（SFT）与指令微调（Instruction Tuning）

- **SFT**：用"输入—理想输出"对，教模型完成特定任务。
- **指令微调**：用多样化"指令—回答"数据，让模型学会**遵循指令**这一元能力。

## 6.4 参数高效微调（PEFT）——当前主流

PEFT 只调整一小部分参数，大幅降低显存与算力。([来源](https://www.cnblogs.com/KubeExplorer/p/18674820))

| 方法 | 机理（概览） | 特点 |
|---|---|---|
| **LoRA** | 冻结原权重，在注意力等层插入低秩矩阵训练 | 主流、即插即用、可多适配器切换 |
| **QLoRA** | 基座 4-bit 量化 + LoRA，引入 NormalFloat | 单卡可微调 65B 级模型 ([来源](https://blog.csdn.net/qq_27590277/article/details/137489646)) |
| **Prefix / Prompt / P-Tuning** | 训练可学习的前缀/软提示向量 | 参数极少，任务专用 |
| **Adapter** | 插入小型适配模块 | 模块化 |

## 6.5 偏好对齐（Alignment）

让模型对齐人类偏好（风格、有用、无害、诚实）。([来源](https://blog.csdn.net/qq_27590277/article/details/137489646))

| 路线 | 机理 | 特点 |
|---|---|---|
| **RLHF（RM + PPO）** | 先训奖励模型，再用强化学习（PPO）优化策略 | 效果强，但流程复杂、不稳定、算力高 |
| **DPO** | 直接用偏好对优化，免去显式奖励模型 | 前沿主流，简单稳定 |
| **其他直接偏好优化** | KTO、ORPO、SimPO 等变体 | 各有取舍 |

> **对齐税（Alignment Tax）**：对齐可能牺牲部分原始能力，需用评测监控能力—安全的平衡。

## 6.6 本层的产物与交付

微调权重（完整或 LoRA 适配器）+ 训练/评测记录 → 交给 L5 做量化与服务化。

# 第 7 章　推理服务与部署适配层（L5）

这是"系统对齐"主战场：把训练好的模型变成**又快、又省、又稳**的在线服务。

## 7.1 三大主线

### 主线一：量化（Quantization）——省显存、提吞吐

- **趋势**：从"只量化权重"走向**全栈量化**（权重 + KV Cache + 激活）。2026 年 **W4A8KV4**（4-bit 权重、8-bit 激活、4-bit KV Cache）成为许多团队默认配置。([来源](https://zhuanlan.zhihu.com/p/2017695401436783377))
- **常见格式**：GPTQ、AWQ（权重）；FP8/INT8（权重+激活）；FP8/INT8/INT4/INT2（KV Cache，不同引擎支持度不同）。
- **KV Cache 量化**：2025 下半年从实验室走向生产，KIVI（2-bit K / 4-bit V）、Gear（残差量化，最高 8× 压缩）等代表方法。([来源](https://zhuanlan.zhihu.com/p/2017695401436783377))
- **权衡**：位宽越低越省，但精度损失越大，需用任务评测把关。

### 主线二：显存与 KV Cache 管理——决定并发与上下文长度

- **PagedAttention（vLLM 核心）**：借鉴操作系统虚拟内存分页，把 KV Cache 切成定长"页"动态分配，显著减少碎片，显存利用率 96%+。([来源](https://www.cnblogs.com/zackstang/p/19036108))
- **前缀缓存 / KV 复用**：相同前缀的请求复用 KV，降低重复计算。
- **企业级 KV 层**：LMCache 等把 KV Cache 作为独立的高效缓存层服务于企业级推理。([来源](https://blog.lmcache.ai/zh/2025/11/24/lmcache面向企业级大语言模型推理的高效kv-cache层/))

### 主线三：服务化与调度——决定吞吐与成本

- **连续批处理（Continuous Batching）**：请求级动态组批，提升 GPU 利用率。
- **投机解码（Speculative Decoding）**：小模型起草、大模型验证，降低延迟。([来源](https://zhuanlan.zhihu.com/p/2017695401436783377))
- **PD 分离（Prefill/Decode Disaggregation）**：把计算密集的 prefill 与访存密集的 decode 拆分调度。
- **分布式推理栈上 K8s**：2025 年起 NVIDIA Dynamo、AIBrix、llm-d、SGLang OME、KServe 等简化推理引擎在 Kubernetes 上的部署与 KV 感知调度。([来源](https://zhuanlan.zhihu.com/p/2017695401436783377))

## 7.2 关键指标（SLA 词汇表）

- **TTFT**（Time To First Token，首 token 延迟）、**TPOT/ITL**（每 token 延迟）、**吞吐（tokens/s、QPS）**、**并发数**、**单位 token 成本**、**P99 延迟**。
- 这些指标是 L5 一切取舍的"裁判"。

# 第 8 章　应用与场景适配层（L6）

把模型嵌入真实业务，产生价值。主线是 **提示工程 → RAG → Agent** 的能力演进。([来源](https://blog.csdn.net/m0_59164304/article/details/140085175))

## 8.1 提示工程（Prompt Engineering）

通过精心设计的提示（含 Few-shot、CoT 思维链、角色设定、输出格式约束）引导模型，无需改权重，是成本最低的适配手段。

## 8.2 检索增强生成（RAG）

- **定义**：2020 年由 Facebook AI Research 提出，先从外部知识库检索相关信息，再据此生成答案，显著提升知识密集型任务的准确性与时效性。([来源](https://zhuanlan.zhihu.com/p/1981389972620735200))
- **典型管线**：文档切分 → 向量化（Embedding）→ 向量库存储 → 检索（向量/关键词/混合）→ 重排（Rerank）→ 拼接上下文 → 生成。
- **价值**：解决"知识时效"与"幻觉"，且无需训练即可注入私有知识——这是与"微调"分工的关键。

## 8.3 智能体（Agent）与 Agentic RAG

- **Agent**：赋予模型自主规划、决策、工具调用（Function Calling）、记忆与反思能力。
- **Agentic RAG**：把 Agent 的"思考—规划"引入 RAG，突破传统 RAG"检索→增强→生成"的被动线性流程，支持工具使用（计算器、搜索、API）、自我反思与纠错。([来源](https://blog.csdn.net/weixin_65514978/article/details/142767579))
- **生态工具**：LangGraph、AutoGPT、CrewAI 等编排框架；MCP 等工具/上下文协议。([来源](https://www.nowcoder.com/discuss/776566610302656512))

## 8.4 Function Calling 与工具生态

RAG 为 Agent 提供动态知识检索；Function Calling 让模型调用外部工具与 API，二者结合构成 Agent 的"手与脑"。([来源](https://blog.csdn.net/weixin_65514978/article/details/142767579))

## 8.5 幻觉治理（贯穿 L6）

斯坦福 2024 研究表明，**RAG + CoT + RLHF 多层协同**可使 LLM 幻觉率较基座降低约 96%——单一技术不够，**多技术协同才是最佳路径**。([来源](https://blog.csdn.net/2401_84204413/article/details/153867077)) 另有实时实体级幻觉检测（线性探针 / LoRA 探针逐 token 检测）用于在线监控。([来源](https://zhuanlan.zhihu.com/p/1977811308687671368))

# 第三篇　横向贯穿能力

这三章不属于某一层，而是**贯穿所有层**的支撑能力。没有它们，纵向各层无法形成闭环。

# 第 9 章　数据工程

- **贯穿性**：预训练语料、CPT 领域语料、SFT 指令数据、偏好对齐数据、RAG 知识库、评测集——每一层适配都吃数据。
- **关键环节**：采集 → 清洗/去重 → 质量过滤 → 脱敏/合规 → 配比（domain mix）→ 标注（人工/合成）→ 版本化。
- **合成数据**：用强模型生成训练/评测数据，是 2025–2026 的重要趋势，但需防"模型坍缩"与偏差放大。
- **核心原则**：**数据质量 > 数据数量**；数据配比直接决定 CPT 的遗忘程度与 SFT 的能力边界。

# 第 10 章　评测、可观测与 LLMOps

## 10.1 评测（Evaluation）

- **通用基准**：MMLU、GSM8K、HumanEval、各类中文榜单等，衡量通用能力。
- **领域/任务评测**：自建贴合业务的评测集才是上线的真正裁判。
- **方法**：自动指标、人工评测、**LLM-as-a-Judge**（用强模型当评委）。
- **专项**：安全/幻觉评测（如 MUCH 多语言声明级幻觉基准）。([来源](https://zhuanlan.zhihu.com/p/1977811308687671368))

## 10.2 LLMOps / 可观测

- **持续集成**：数据—训练—评测—部署的自动化流水线；模型/数据/提示三者都要版本化。
- **在线监控**：延迟、成本、质量漂移、幻觉率、安全事件的实时观测与告警。
- **回滚与灰度**：A/B、灰度发布、影子流量，确保新模型/新提示可控上线。

> **闭环思想**：适配不是一次性交付，而是"评测发现差距 → 定位到某层 → 改进 → 再评测"的持续循环。第 12 章的 Pipeline 以此为骨架。

# 第 11 章　安全、合规与成本治理

- **安全**：输入/输出护栏（Guardrails）、越狱防护、敏感内容过滤、Red Teaming。
- **合规**：数据隐私、算法备案、可审计性、行业监管（金融/医疗尤甚）、国产化/自主可控要求。
- **成本治理**：模型分级路由（小模型兜底、大模型兜难）、缓存、量化、按 SLA 选硬件——把"质量×延迟×成本"三角显式管理起来。

# 第四篇　流程与方法论

# 第 12 章　端到端适配 Pipeline

把六层栈展开成一条可执行的工程流水线：

```
① 需求与可行性
   └─ 明确任务、SLA（延迟/并发/成本）、合规约束、四个对齐里以哪个为主

② 基座选型
   └─ 开源 vs 闭源 API、参数规模、上下文长度、license、国产化要求
       └─ 决策：能用 API + 提示/RAG 解决吗？能 → 直接跳到 ⑥

③ 数据工程（横向第 9 章）
   └─ 领域语料 / 指令数据 / 偏好数据 / RAG 知识库 / 评测集

④ 训练侧适配（L4，第 6 章）
   └─ 需要新知识？→ CPT；需要新任务/格式？→ SFT/PEFT；需要对齐偏好？→ DPO/RLHF

⑤ 模型与引擎适配（L1–L3，第 3–5 章）
   └─ 选硬件 → 选引擎 → 必要时改结构/写算子 → 验证功能正确

⑥ 推理服务化（L5，第 7 章）
   └─ 量化（W4A8KV4 等）→ KV/显存管理 → 批处理/投机解码 → 服务化上 K8s

⑦ 应用编排（L6，第 8 章）
   └─ 提示 / RAG / Agent / 工具 / 工作流

⑧ 评测与上线（横向第 10 章）
   └─ 离线评测达标 → 灰度 → 监控 → 反馈
                              │
   ┌──────────────────────────┘
   └─ 回到 ③/④/⑥/⑦ 中"差距所在的那一层"迭代  ← 闭环
```

> **关键**：步骤 ② 的分支决定了项目"重量级"。很多项目根本不需要 ④（不微调），直接走 ②→⑥→⑦。

# 第 13 章　适配决策树与选型方法论

## 13.1 "该在哪一层动手"决策树

```
问题：模型表现不满足需求
│
├─ 是"不会按要求格式/语气输出"？
│     → 先试【提示工程】(L6)；仍不行 → 【SFT/指令微调】(L4)
│
├─ 是"不知道某些私有/最新知识"？
│     → 【RAG】(L6) 优先；知识极大且需内化 → 【CPT】(L4)
│
├─ 是"领域专业能力整体偏弱"？
│     → 【领域SFT/PEFT】(L4)，必要时 + 【CPT】
│
├─ 是"输出不安全/不符合价值观"？
│     → 【对齐 DPO/RLHF】(L4) + 【护栏】(L11)
│
├─ 是"太慢/太贵/并发不够"？
│     → 【量化 + KV管理 + 批处理/投机解码】(L5)；再不行 → 换硬件/引擎(L1/L2)
│
└─ 是"在目标芯片上跑不起来/跑不快"？
      → 【硬件/引擎/结构适配】(L1–L3)：算子覆盖、精度、并行
```

## 13.2 选型核心权衡表

| 维度 | 偏向"轻量适配"（提示/RAG/PEFT） | 偏向"重量适配"（CPT/全参/换硬件） |
|---|---|---|
| 数据量 | 少 | 大（领域语料 10B+ tokens） |
| 知识时效性要求 | 高（选 RAG） | 低（可内化） |
| 成本/工期 | 低 | 高 |
| 可控性/私有化 | 中 | 高 |
| 灾难性遗忘风险 | 低 | 高（需 replay/配比） |

## 13.3 三条经验法则

1. **自底向上验证，自顶向下设计**：设计从业务（L6）倒推需求，实施从硬件（L1）向上验证可行性。
2. **先低强度后高强度**：提示 < RAG < PEFT < 全参 < CPT，能浅则浅。
3. **评测先行**：动手前先建评测集，否则无法判断任一层的改动是否真的有效。

# 第 14 章　工程实践与避坑指引

- **坑 1：层错位**。用提示硬解领域能力缺失，或用微调硬解知识时效问题——选错层，事倍功半。（对策：用第 13 章决策树。）
- **坑 2：CPT 灾难性遗忘**。只喂领域语料导致通用能力崩塌。（对策：混合通用语料 + replay + 控制学习率。([来源](https://blog.csdn.net/sinat_37574187/article/details/144110788))）
- **坑 3：量化精度悬崖**。盲目压到极低位宽导致任务质量骤降。（对策：用业务评测集逐位宽验证，W4A8KV4 通常是较稳的折中。）
- **坑 4：只看通用榜单**。基准分高 ≠ 业务可用。（对策：自建领域评测 + LLM-as-Judge + 人工抽检。）
- **坑 5：硬件适配低估工时**。国产芯片算子覆盖/工具链成熟度差异大。（对策：先做算子覆盖率与小规模实测 POC 再承诺工期。）
- **坑 6：Agent 过度工程**。能用 RAG 解决的非要上多智能体，徒增延迟与不可控。（对策：从提示→RAG→单 Agent→多 Agent 逐级升级。）
- **坑 7：忽视闭环**。一次性交付不留监控与回滚。（对策：上线即接入 LLMOps 观测与灰度。）

# 第 15 章　趋势、局限与待补充方向

## 15.1 趋势（2025–2026）

1. **硬件原生协同**：从"模型适配芯片"走向"芯片与模型协同设计"（DeepSeek-V4 写入昇腾 NPU 是标志）。([来源](https://www.nbd.com.cn/articles/2026-04-24/4358607.html))

3. **对齐轻量化**：DPO 及其变体替代复杂 RLHF。
4. **应用 Agent 化**：从 RAG 走向 Agentic RAG / 多智能体编排。
5. **推理服务云原生化**：Dynamo/llm-d/AIBrix/KServe 把推理栈标准化到 K8s。([来源](https://zhuanlan.zhihu.com/p/2017695401436783377))

## 15.2 本书的局限性（诚实说明）

- **不展开单点细节**：本书是地图，不是某一层的施工手册。具体到"如何写一个昇腾算子""LoRA 秩怎么选""PagedAttention 内核实现"，需另读专题资料。
- **时效性**：硬件路线图、引擎特性、量化方案演进极快，本书数据截至 2026 年中，落地前请核对最新版本。
- **来源构成**：本报告综合了行业媒体、技术社区（知乎/CSDN/博客园）与厂商资料，**部分技术社区内容未经严格学术同行评审**，关键决策请以官方文档与一手实测为准。
- **多模态/具身等未充分展开**：本书以文本 LLM 适配为主轴，视觉/语音/多模态/具身智能的适配仅在 L3 略提，可作为待补充方向。

## 15.3 建议的下一步深读方向

1. 某一层的专题手册（如昇腾 CANN 算子开发、vLLM 模型接入指南）。
2. 评测体系建设（领域评测集 + LLM-as-Judge 工程化）。
3. 成本/SLA 建模（把"质量×延迟×成本"三角量化为可决策模型）。
